QR Code & Contact Card Generator

Technical documentation for the handwritten JavaScript encoder

Overview

The generator in QRCodeGenerator.html creates QR codes entirely in the browser. It does not call an external QR service and does not depend on QRCode.js or any third-party library. The encoder builds a QR matrix in JavaScript, then renders the result as SVG.

The tool supports two payload types: normal URL text and vCard contact data. The same QR encoder is used for both; only the payload-building step changes.

Design goal: keep the implementation self-contained, readable, inspectable, and exportable as SVG or PNG.

Theoretical Background

What a QR Code Contains

A QR code is a square matrix of dark and light modules. Some modules are structural patterns that help scanners locate and interpret the code; the rest store encoded data and Reed-Solomon error correction bytes.

  • Finder patterns: the large square markers in three corners.
  • Timing patterns: alternating modules used to determine grid spacing.
  • Alignment patterns: smaller markers used in larger versions to correct distortion.
  • Format information: error correction level and mask pattern.
  • Version information: required for QR versions 7 and above.
  • Data and ECC modules: the encoded payload and error correction bytes.

Versions and Capacity

QR versions control matrix size. Version 1 is 21 by 21 modules. Each higher version adds four modules per side. This implementation supports versions 1 through 10 at error correction level L.

URLs usually fit in smaller versions. vCards are longer because they include multiple structured lines, so the encoder may choose a larger version such as Version 8.

Error Correction

QR codes use Reed-Solomon error correction over GF(256). This lets scanners recover the payload even if part of the QR code is damaged, blurred, or visually imperfect.

Implementation Pipeline

  1. Build payload: read the selected form mode and produce either a URL string or a vCard string.
  2. Select QR version: choose the smallest supported version that can hold the UTF-8 byte payload.
  3. Encode byte mode: write the QR byte-mode indicator, payload length, UTF-8 bytes, terminator, and pad bytes.
  4. Create data blocks: split data into QR-standard blocks for larger versions.
  5. Generate ECC: calculate Reed-Solomon bytes for each block.
  6. Interleave codewords: interleave data and ECC bytes in the order expected by scanners.
  7. Draw function patterns: place finder, timing, alignment, dark module, format reservations, and version reservations.
  8. Place data bits: write the bitstream in the QR zigzag pattern from bottom-right upward.
  9. Apply masks: try all eight QR masks.
  10. Score masks: choose the mask with the lowest QR penalty score.
  11. Render SVG: convert dark modules into SVG rectangles with a quiet zone.

vCard Payload

When Contact / vCard mode is selected, the script builds a valid vCard 3.0 payload. Most mobile QR scanners recognize this format and offer to save it as a contact. 

BEGIN:VCARD
VERSION:3.0
N:Ahmed;Dina;;;
FN:Dina Ahmed
TEL;TYPE=CELL:+201022310450
EMAIL:dinaahmedhafez655@gmail.com
TITLE:Data Science Student
ORG:Alexandria University
END:VCARD

The implementation uses CRLF line endings (\r\n) because vCard parsers expect that convention. Special characters such as commas, semicolons, backslashes, and line breaks are escaped before encoding.

Key Functions

  • buildVCardPayload(): validates contact fields and creates the vCard text.
  • getCurrentQrText(): chooses URL or vCard payload based on the selector.
  • encodeByteData(text, version): converts payload text into QR byte-mode data codewords.
  • reedSolomonRemainder(data, eccLength): generates ECC bytes for one block.
  • makeDataBlocks() and interleaveBlocks(): prepare larger QR versions correctly.
  • buildFunctionPatterns(version): draws and reserves all fixed QR structures.
  • drawData(matrix, dataBits, mask): places payload bits in QR zigzag order.
  • penaltyScore(modules): scores each mask candidate.
  • buildQrSvg(text): creates the final SVG element.
  • exportQrSvg() and exportQrPng(): download the current QR output.

Implementation Choices

Why SVG?

SVG is sharp at any size, easy to export, and ideal for QR codes because each module can be drawn as a precise rectangle. The file also uses shape-rendering: crispEdges to avoid blurry module edges.

Why Byte Mode?

URLs and vCards contain punctuation, symbols, and sometimes non-ASCII characters. Byte mode with UTF-8 is the simplest reliable encoding choice for those payloads.

Why Error Correction Level L?

Level L gives the most data capacity. This matters for vCards, which can be long. For a plain black-and-white QR code with no logo overlay, Level L is usually acceptable.

Why Support Version 10?

Small URL QR codes fit in early versions, but contact cards may require larger matrices. Version 10 gives enough room for realistic vCard payloads while keeping the handwritten implementation manageable.

Validation and Debugging

The script includes internal self-tests for format bits, GF(256) multiplication, and known small matrices. It also exposes debugging helpers in the browser console. 

window.exportQrMatrixText()
window.exportQrMatrixSignature()

If a QR code does not scan, the first things to inspect are payload length, QR version, alignment pattern placement, version information bits, and whether the rendered SVG is visually crisp.

Limitations

  • The encoder supports QR versions 1-10 only.
  • It uses byte mode only.
  • It uses error correction level L only.
  • Very long vCards may exceed the supported capacity.
  • No logo embedding or styling overlays are applied, which helps preserve scannability.
